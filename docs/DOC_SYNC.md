# Documentation Sync Rules

这份文档说明：代码变化后，什么时候需要同步文档，以及如何避免文档被过度修改。

目标是防止两种问题：

- 代码变了，文档没变，导致文档漂移。
- 文档被大幅重写，导致原有架构规则被误改。

## 必须同步文档的情况

如果代码改动满足下面任一条件，必须检查并更新相关文档。

### 项目结构变化

- 新增模块
- 删除模块
- 移动模块
- 改变目录结构
- 改变文件职责

通常需要更新：

- `docs/ARCHITECTURE.md`
- `docs/AGENT_GUIDE.md`

### 模块依赖变化

- 某个模块新增对另一个模块的依赖
- 依赖方向改变
- 新增共享类型
- 移除或替换核心模块

通常需要更新：

- `docs/AGENT_GUIDE.md` 的依赖方向规则
- `docs/ARCHITECTURE.md` 的模块说明

### 用户可见行为变化

- 新增命令
- 删除命令
- 改变命令用法
- 改变输出格式
- 改变运行方式
- 新增环境变量
- 改变 API Key 配置方式

通常需要更新：

- `README.md`

### Agent 主流程变化

- 改变 AI 自主工具调用流程
- 改变工具调用轮次限制
- 改变普通对话进入 LLM 的方式
- 改变消息写入上下文的时机
- 改变系统 prompt 构建方式

通常需要更新：

- `docs/ARCHITECTURE.md`
- `docs/AGENT_GUIDE.md`

### Tool 行为变化

- 新增工具
- 删除工具
- 改变工具名称
- 改变工具输入格式
- 改变工具安全边界
- 改变工具是否能访问文件/网络

通常需要更新：

- `README.md`
- `docs/ARCHITECTURE.md`
- `docs/AGENT_GUIDE.md`

### Memory / Todo / Skill / Sub-Agent / Hook 行为变化

- 新增命令
- 改变存储路径
- 改变数据格式
- 改变自动触发时机
- 改变启用方式
- 改变职责边界

通常需要更新：

- `README.md`
- `docs/ARCHITECTURE.md`
- `docs/AGENT_GUIDE.md`

### 安全边界变化

- 文件工具允许访问更大范围
- 新增写文件能力
- 新增删除文件能力
- 新增执行系统命令能力
- 新增网络请求能力
- 改变密钥读取方式

通常需要更新：

- `README.md`
- `docs/AGENT_GUIDE.md`
- 后续如有 `docs/SECURITY.md`，也要更新

## 不需要同步文档的情况

下面这些通常不需要改文档：

- 代码格式化
- 局部变量重命名
- 提取小函数但不改变模块职责
- 修复 bug 但不改变用户可见行为
- 优化错误信息但不改变命令用法
- 改善内部实现但不改变依赖方向
- 增加注释
- 调整测试或构建缓存

如果不确定，优先只在最终说明中写：

```text
文档无需更新，因为本次改动没有改变模块职责、命令用法或用户可见行为。
```

## 文档修改原则

更新文档时遵守：

1. 只改与本次代码变化直接相关的段落。
2. 不要重写整份文档。
3. 不要顺手润色无关内容。
4. 不要改变已有架构原则，除非代码变化真的改变了架构。
5. 不要删除仍然准确的说明。
6. 新增内容要短，优先链接到更详细的专门文档。
7. 如果一个主题开始膨胀，单独拆出新的 `docs/*.md`。

## 代码改动后的检查清单

每次代码改动后，问自己：

```text
1. 是否新增、删除或移动模块？
2. 是否改变模块职责？
3. 是否改变模块依赖方向？
4. 是否新增、删除或改变命令？
5. 是否新增或改变工具？
6. 是否改变文件、网络或密钥安全边界？
7. 是否改变 Memory/Todo/Skill/Sub-Agent/Hook 行为？
8. 是否改变数据文件路径或格式？
9. 是否改变用户运行方式？
10. 是否改变 Agent 主流程？
```

如果全部为“否”，通常不需要更新文档。

如果任意一项为“是”，更新对应文档的最小相关段落。

## AI Assistant 规则

AI assistant 修改本项目时，必须：

1. 先阅读根目录 `AGENT_GUIDE.md`。
2. 再阅读 `docs/AGENT_GUIDE.md` 和 `docs/ARCHITECTURE.md`。
3. 改代码后按本文件的检查清单判断是否需要更新文档。
4. 如果需要更新文档，只做最小相关修改。
5. 如果不需要更新文档，在最终回复里说明原因。
6. 不要为了“看起来更完整”而重写大段文档。
7. 不要在没有代码依据的情况下修改架构规则。

## 推荐文档分工

```text
README.md
  面向使用者：怎么运行、怎么用、有哪些命令。

docs/ARCHITECTURE.md
  面向开发者：模块结构、主流程、数据流。

docs/AGENT_GUIDE.md
  面向人类和 AI：协作规则、模块边界、扩展流程。

docs/DOC_SYNC.md
  面向维护者和 AI：代码变化后如何同步文档。
```

后续可以继续增加：

```text
docs/ROADMAP.md
docs/SECURITY.md
docs/ADR/
```
